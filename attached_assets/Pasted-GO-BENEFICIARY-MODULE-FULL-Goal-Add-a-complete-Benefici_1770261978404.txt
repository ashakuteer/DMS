GO BENEFICIARY MODULE (FULL)

Goal:
Add a complete Beneficiaries module for Asha Kuteer Foundation with profiles, photo upload, updates/timeline, and sponsor linking (Donor ↔ Beneficiary). This supports 3 homes: Orphan Girls Home, Visually Challenged Boys Home, Old Age Home.

A) Database / Prisma
1) Create Beneficiary model:
- id (uuid)
- code (unique string like AKF-BEN-000001)
- fullName (string, required)
- homeType (enum: ORPHAN_GIRLS, BLIND_BOYS, OLD_AGE)
- gender (enum: MALE, FEMALE, OTHER, optional)
- dobDay (int 1-31, optional)
- dobMonth (int 1-12, optional)
- approxAge (int, optional)  // for seniors where DOB unknown
- joinDate (date, optional)
- educationClassOrRole (string, optional) // “5th Class”, “Retired”, “Caregiver note”
- schoolOrCollege (string, optional)
- healthNotes (string, optional)
- background (text, optional)
- hobbies (string, optional)
- dreamCareer (string, optional)
- favouriteSubject (string, optional)
- favouriteGame (string, optional)
- favouriteActivityAtHome (string, optional)
- bestFriend (string, optional)
- sourceOfPrideOrHappiness (string, optional)
- funFact (string, optional)
- additionalNotes (text, optional)
- protectPrivacy (boolean default false)
- photoUrl (string, optional)
- status (enum: ACTIVE, INACTIVE default ACTIVE)
- createdAt, updatedAt

2) Create Sponsorship model (link donors to beneficiaries):
- id (uuid)
- donorId (fk)
- beneficiaryId (fk)
- sponsorshipType (enum: FULL, PARTIAL, EDUCATION, MEDICAL, FOOD, GROCERIES, MONTHLY_SUPPORT, ONE_TIME)
- amount (number optional)  // when monetary
- inKindItem (string optional) // “Rice bag”, “Medicines”
- frequency (enum: ONE_TIME, MONTHLY, QUARTERLY, YEARLY, ADHOC default ADHOC)
- startDate (date optional)
- endDate (date optional)
- notes (text optional)
- isActive (boolean default true)
- createdAt, updatedAt

Constraints:
- One donor can sponsor multiple beneficiaries.
- One beneficiary can have multiple sponsors (for partial sponsorship).
- Prevent duplicate active sponsorship rows with same donorId+beneficiaryId+sponsorshipType.

3) Create BeneficiaryUpdate model:
- id (uuid)
- beneficiaryId (fk)
- title (string)
- content (text)
- mediaUrls (json/text array optional)  // store as JSON string if needed
- createdByUserId (fk)
- createdAt

4) Create BeneficiaryTimelineEvent model (optional if you prefer reusing updates):
- id (uuid)
- beneficiaryId
- eventType (enum: JOINED, HEALTH, EDUCATION, ACHIEVEMENT, GENERAL)
- eventDate (date)
- description (text)
- createdAt

Migrate DB + generate Prisma client.

B) Backend APIs (Nest)
Create module: beneficiaries
Routes:
1) GET /api/beneficiaries
- filters: homeType, status, search (name/code), sponsored (true/false)
- pagination: page, limit
Return list with: id, code, fullName, homeType, photoUrl, status, dobMonth/dobDay, approxAge, activeSponsorsCount.

2) POST /api/beneficiaries
Create beneficiary (validate required fullName + homeType). Auto-generate code AKF-BEN-000001 style.

3) GET /api/beneficiaries/:id
Return full beneficiary profile + sponsors + recent updates + timeline stats.

4) PATCH /api/beneficiaries/:id
Update fields.

5) POST /api/beneficiaries/:id/photo
Photo upload:
- Use existing storage approach used for donor photos if present.
- If no storage exists, implement simple upload to Supabase Storage (since SUPABASE_URL and SERVICE_ROLE exist).
Return photoUrl and save to beneficiary.

6) Sponsors APIs:
- GET /api/beneficiaries/:id/sponsors
- POST /api/beneficiaries/:id/sponsors  (link donor)
- PATCH /api/sponsorships/:id (edit)
- DELETE /api/sponsorships/:id (unlink/disable)

Also add donor side:
- GET /api/donors/:id/sponsorships

7) Updates APIs:
- GET /api/beneficiaries/:id/updates
- POST /api/beneficiaries/:id/updates
- DELETE /api/beneficiary-updates/:id
